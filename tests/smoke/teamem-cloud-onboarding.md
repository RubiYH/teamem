# Teamem Cloud Onboarding Smoke

This manual smoke verifies the managed cloud path from web login through a working Teamem runtime. It must pass without asking the user to self-host Docker.

## Prerequisites

- Teamem Cloud web app deployed or running locally with the env from `docs/deploy/teamem-cloud.md`.
- Supabase control-plane migration applied.
- GitHub OAuth configured for the active web origin. Google OAuth may be enabled as an optional second provider.
- Hosted Teamem runtime, or a local Teamem runtime for development smoke, reachable at `TEAMEM_CLOUD_RUNTIME_URL`.
- Matching `TEAMEM_CLOUD_RUNTIME_PROVISIONING_TOKEN` configured in the web app and runtime.
- `teamem` CLI installed in the shell used for the join step.

## Story 1: Login

Action:

1. Open the Teamem Cloud landing page.
2. Choose the GitHub login action.
3. Complete the OAuth provider flow.

Expected:

- The browser returns to the Teamem Cloud dashboard.
- The dashboard shows the signed-in account state.
- No Docker or VPS setup instructions are required.

## Story 2: Create One Free Space

Action:

1. From the dashboard, create a free Space.
2. Wait for provisioning to complete.

Expected:

- Exactly one active free Space is shown for the account.
- The Space has a display name, runtime server URL, and room code metadata.
- The setup command is visible after provisioning succeeds.

## Story 3: Copy Setup Command

Action:

1. Copy the setup command shown for the Space.
2. Confirm it has this shape:

```bash
teamem init --join --server-url <runtime-url> --room-code <room-code> --member-name <your-name>
```

Expected:

- `<runtime-url>` matches `TEAMEM_CLOUD_RUNTIME_URL`.
- `<room-code>` is an opaque room code generated by the runtime.
- The command does not mention `docker compose`, a VPS, or self-hosted runtime setup.

## Story 4: CLI Join

Action:

1. In a clean local repository or test checkout, run the copied command with a real member name:

```bash
teamem init --join --server-url <runtime-url> --room-code <room-code> --member-name alice
```

Expected:

- The CLI joins the cloud-provisioned Space.
- Local Teamem credentials/config are written for that repository.
- The CLI does not require Docker to be installed or a local Teamem server to be started.

## Story 5: Claude Code Runtime Usability

Action:

1. Start Claude Code through the Teamem CLI:

```bash
teamem cc
```

2. In Claude Code, request a Teamem status or briefing check.
3. Claim a small test scope or call a basic Teamem runtime read path such as the briefing/status surface.

Expected:

- Claude Code can reach the configured Teamem runtime.
- Teamem reports the joined Space and current member identity.
- A basic coordination operation works against the cloud-provisioned Space.
- No Docker self-hosting step appears anywhere in the cloud onboarding path.

## Cleanup

Action:

1. Delete the test Space from the dashboard.
2. Confirm the dashboard returns to an empty or create-ready state.

Expected:

- The free quota is recovered after delete completes.
- A new free Space can be created later by the same account.

## Manual Provider Steps Still Required

- Vercel env configuration and deployment.
- Supabase project creation, migration, and credential copy.
- GitHub OAuth app setup and optional Google OAuth setup.
- Runtime deployment or local runtime startup for development smoke.
- Shared runtime provisioning-token generation and installation in both places.
